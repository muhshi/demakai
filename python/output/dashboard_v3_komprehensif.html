Metode 02 ⭐
Hybrid Search

SQL LIKE + Semantic Search (embedding Gemini) — tanpa preprocessing, tanpa contoh lapangan.

Menggabungkan dua jalur: (1) SQL LIKE untuk kecocokan leksikal, dan (2) vector search menggunakan cosine distance ke embedding Gemini. Hasil digabung dan diberi boost berdasarkan lokasi match (contoh_lapangan > judul > deskripsi). Tanpa preprocessing, query masuk apa adanya — namun semantic embedding sudah mampu memahami intent query informal.

Metode 03
SQL LIKE + Preprocessing

SQL LIKE dengan query expansion (sinonim + variasi KBLI/KBJI) — tanpa semantic search.

Preprocessing meliputi: lowercase, cleaning, stopword removal, stemming, dan query expansion (sinonim dan variasi domain KBLI/KBJI). Contoh: "kontruksi rumah" → "bangunan hunian konstruksi rumah | +KBLI: jasa bangun rumah, proyek bangun rumah". SQL LIKE kemudian mencari frasa yang diperkaya ini.

🌑 2. Semantic Search = Terobosan Utama
Hybrid Search (MRR KBLI=0.9167) meningkat 33× dibanding Baseline. Gemini embedding memahami intent query: "RESOLES" → industri makanan, "MATON" → pertanian padi. Ini adalah komponen terpenting sistem — penambahan fitur lain hanya signifikan jika tidak merusak kualitas embedding ini.
✂ 3. Preprocessing: Untung untuk SQL, Rugi untuk Hybrid (Sebelum Fix)
Preprocessing meningkatkan SQL LIKE dari MRR=0.028 ke 0.8944 (+3117%). Namun, Hybrid + Preprocessing (MRR=0.8500) justru lebih rendah dari Hybrid Raw (0.9167). Penyebab: expanded tokens 25+ kata menambah noise ke vektor embedding, dan boost distance override paksa (0.04) menyebabkan false-positive.
✅ 4. Perbaikan Sistem: 3 Fix = Performa Optimal
Tiga perbaikan kode yang diimplementasikan: (1) Boost proporsional bukan override tetap, (2) Embed dari query original bukan teks expansion, (3) Core tokens (bukan expanded) untuk matching. Hasilnya: KBLI MRR 0.8500 → 0.9278 dan KBJI MRR 0.6861 → 0.8556. Hybrid+Prep+CL kini menjadi metode TERBAIK.
📋 5. Contoh Lapangan: Efektif Setelah Fix
Sebelum fix: field contoh_lapangan tidak memberikan peningkatan nyata karena boost override-nya menyebabkan false-positive. Setelah fix dengan boost proporsional: contoh lapangan memberikan sinyal yang tepat — item yang benar-benar match secara semantik dan textual mendapat prioritas lebih tinggi tanpa mengorbankan item relevan lainnya.

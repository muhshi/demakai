Metode 02 ⭐ TERBAIK
Hybrid Search

Menggabungkan SQL LIKE (lexical) + Semantic Search (embedding Gemini) — tanpa preprocessing, tanpa contoh lapangan.

Hybrid Search menggabungkan dua pendekatan: (1) SQL LIKE untuk pencocokan leksikal pada judul/deskripsi, dan (2) vector search menggunakan embedding Gemini untuk pemahaman semantik. Hasil kedua pendekatan digabung dengan boosting berbasis lokasi match. Tanpa preprocessing, query langsung digunakan apa adanya — namun kemampuan semantik embedding sudah mampu memahami intent query informal.

Metode 03
SQL LIKE + Preprocessing

SQL LIKE dengan Query Expansion (sinonim + variasi KBLI/KBJI) — tanpa semantic search.

Preprocessing meliputi: lowercase, cleaning, stopword removal, stemming, dan query expansion (penambahan sinonim dan variasi kata terkait konteks KBLI/KBJI). Hasilnya digunakan untuk pencarian dengan SQL LIKE. Contoh: "cabut rumput sawah" → "liar maton mencabut pertanian rumput sawah tanaman pangan | +KBLI: lahan tani padi, sawah padi, usaha tani"

📊 Analisis Perbandingan Metode

Evaluasi menggunakan 60 query (30 KBLI 2025 + 30 KBJI 2014) yang konsisten di semua metode. Metode terbaik berdasarkan MRR: Hybrid Search (M2) — MRR = 0.8610.

🔍 Temuan Utama: SQL LIKE Murni Tidak Efektif
SQL LIKE Baseline (MRR=0.014) hampir tidak berguna untuk query informal pengguna lapangan. Dari 60 query, hanya 2 query menghasilkan hit — dan keduanya hanya karena kata kunci yang terlalu generik ("PERTANIAN PADI", "PENYEDIAAN MAKANAN"). Query informal seperti "MATON", "KLONTONG", "SIOMAY" tidak cocok dengan teks KBLI yang formal.
🌑 Hybrid Search: Terobosan Terbesar
Lompatan dari SQL LIKE (MRR=0.014) ke Hybrid Search (MRR=0.861) merupakan peningkatan tertinggi — sekitar 62× lipat. Semantic search (embedding Gemini) mampu memahami intent query seperti "KONTRUKSI" → konstruksi bangunan, "RESOLES" → industri makanan, tanpa membutuhkan kata kunci eksak.
✂ Dampak Preprocessing: Trade-off
Preprocessing dengan query expansion meningkatkan SQL LIKE secara signifikan (+6.000%), namun menurunkan Hybrid Search (dari 0.861 → 0.768). Penyebab: teks yang terlalu panjang akibat expansion menciptakan noise dalam representasi vektor embedding. Ini disebut over-expansion problem.
📋 Peran Field Contoh Lapangan
Field contoh_lapangan berisi deskripsi aktivitas nyata dalam bahasa informal. Penambahan field ini tidak mengubah metrik M5 vs M3 (keduanya MRR=0.838) karena SQL LIKE sudah mencakupnya sejak awal. Namun, field ini vital untuk transparansi: setiap rekomendasi dapat dijelaskan dengan contoh nyata yang relevan.
🔬 Kesenjangan KBLI vs KBJI
Pada semua metode, KBLI 2025 konsisten lebih mudah ditemukan (~10-15% lebih tinggi dari KBJI 2014). Ini karena deskripsi KBLI 2025 lebih kaya (updated) dan lebih banyak contoh lapangan yang tersedia. KBJI 2014 memiliki kode jabatan yang lebih abstrak dan kurang representatif dalam data embedding.
🏆 Rekomendasi Praktis
Produksi: Gunakan Hybrid Search murni (M2) untuk performa terbaik (MRR=0.861). Fallback: SQL LIKE + Preprocessing (M3) jika API embedding tidak tersedia. Presentasi/Penelitian: Sistem FINAL (M6) untuk transparansi dan demonstrasi pipeline lengkap.
